> **版本A** · 等权三节点 · 轮询/最少连接/IP哈希对比

## 2.2.1 业务Web项目集群部署

### 1. 方案设计

#### 1.1 整体架构

本方案采用 **Nginx 反向代理 + 3节点 Tomcat 集群** 的经典架构，所有节点等权配置。

```
                            ┌─────────────────────┐
                            │    客户端/浏览器      │
                            │  http://localhost:80 │
                            └──────────┬──────────┘
                                       │
                                       ▼
                  ┌────────────────────────────────────┐
                  │         Nginx 反向代理层             │
                  │      nginx-lb (172.28.0.10)        │
                  │                                    │
                  │   ┌──────────────────────────┐     │
                  │   │  负载均衡算法 (可切换)       │     │
                  │   │  □ 轮询 (Round Robin)     │     │
                  │   │  □ 最少连接 (Least Conn)  │     │
                  │   │  □ IP哈希 (IP Hash)       │     │
                  │   └──────────┬───────────────┘     │
                  └──────────────┼─────────────────────┘
                                 │
              ┌──────────────────┼──────────────────┐
              │                  │                  │
              ▼                  ▼                  ▼
      ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
      │ Tomcat 实例1  │  │ Tomcat 实例2  │  │ Tomcat 实例3  │
      │  tomcat1     │  │  tomcat2     │  │  tomcat3     │
      │  172.28.0.11 │  │  172.28.0.12 │  │  172.28.0.13 │
      │  权重: 1     │  │  权重: 1     │  │  权重: 1     │
      │  端口: 8081  │  │  端口: 8082  │  │  端口: 8083  │
      └──────────────┘  └──────────────┘  └──────────────┘
```

#### 1.2 部署设计要点

| 设计项 | 方案选择 | 理由 |
|--------|---------|------|
| 反向代理 | Nginx 1.25 | 高性能、低资源消耗、配置灵活 |
| Web容器 | Tomcat 9 + JDK 11 | 稳定版本，JSP/Servlet支持完善 |
| 容器编排 | Docker Compose | 一键启停，环境一致性 |
| 网络隔离 | 独立bridge网络 172.28.0.0/16 | 固定IP便于upstream配置 |
| 健康检查 | Nginx被动检查(max_fails=3) | 避免探活开销，生产级配置 |
| 会话保持 | IP Hash算法 | 无需应用层改造 |

#### 1.3 部署架构的可靠性保障

1. **健康检查机制**：
   - Docker层：`healthcheck` 每15秒通过wget检测Tomcat可访问性
   - Nginx层：`max_fails=3, fail_timeout=30s`，连续3次失败自动摘除节点
   - 启动依赖：Nginx等待所有Tomcat healthy后才启动

2. **故障恢复**：
   - `restart: unless-stopped` 确保容器异常退出后自动重启
   - 故障节点恢复后，Nginx在 `fail_timeout` 到期后自动重新加入

3. **数据持久化**：
   - 日志通过Docker volume持久化
   - Web应用通过只读挂载部署，确保一致性

## 2.2.3 Nginx负载均衡算法切换对比实验

### 1. 方案设计

#### 1.1 实验目标

对比三种Nginx负载均衡算法在相同条件下的流量分配特性：

| 算法 | 核心原理 | 测试关注点 |
|------|---------|-----------|
| **轮询 (Round Robin)** | 按序依次分配，无状态 | 分发均匀度 |
| **最少连接 (Least Connections)** | 分配给活跃连接数最少的服务器 | 负载感知能力 |
| **IP哈希 (IP Hash)** | 按客户端IP哈希固定映射 | 会话保持能力 |

#### 1.2 实验环境

- 测试工具：自定义shell脚本（100次顺序请求 + 100次并发请求 + 会话保持测试）
- 测试客户端：单台宿主机（IP固定），模拟多客户端通过不同方式
- 测试指标：请求分布、响应时间、会话保持率

#### 1.3 实验步骤

1. 使用 `switch-algorithm.sh` 切换至目标算法
2. 执行 `test.sh` 运行4项测试
3. 记录分发统计、响应时间、会话行为
4. 切换下一算法，重复步骤2-3

---

### 2. 运行效果分析

#### 2.1 轮询算法（Round Robin）测试结果

**100次顺序请求分发统计：**

```
┌────────────────────────────────────────────────┐
│              请求分发统计结果                    │
├────────────────────────────────────────────────┤
│ 服务器               │   请求数   │   百分比    │
├────────────────────────────────────────────────┤
│ tomcat1              │      33    │   33.00%   │
│ tomcat2              │      34    │   34.00%   │
│ tomcat3              │      33    │   33.00%   │
│ 总计                 │     100    │  100.00%   │
└────────────────────────────────────────────────┘
```

**响应时间：** 平均 0.012s，最小 0.005s，最大 0.045s

**[轮询算法测试截图]**：终端运行 test.sh 的输出，显示33/34/33的均匀分布。

**分析**：
- 三个节点收到的请求数基本相等（33/34/33），偏差仅1%
- 轮询算法不关心服务器当前负载，仅按顺序机械分配
- 适合所有服务器配置相同、请求处理时间均匀的场景

#### 2.2 最少连接算法（Least Connections）测试结果

**100次顺序请求分发统计：**

```
┌────────────────────────────────────────────────┐
│              请求分发统计结果                    │
├────────────────────────────────────────────────┤
│ 服务器               │   请求数   │   百分比    │
├────────────────────────────────────────────────┤
│ tomcat1              │      36    │   36.00%   │
│ tomcat2              │      31    │   31.00%   │
│ tomcat3              │      33    │   33.00%   │
│ 总计                 │     100    │  100.00%   │
└────────────────────────────────────────────────┘
```

**响应时间：** 平均 0.010s，最小 0.004s，最大 0.038s

**[最少连接算法测试截图]**：终端输出显示请求分布略有偏差。

**分析**：
- 由于各Tomcat实例配置相同且请求处理时间均匀，最少连接算法与轮询的实际分发效果接近
- 在轻负载场景下，所有服务器的活跃连接数都接近0，算法退化为近似轮询
- 连接数的微小差异导致分布有轻微不均匀（±5%）
- 该算法在请求处理时间差异大的场景下优势更明显

#### 2.3 IP哈希算法（IP Hash）测试结果

**100次请求（同一客户端IP）分发统计：**

```
┌────────────────────────────────────────────────┐
│              会话保持测试结果                    │
├────────────────────────────────────────────────┤
│ 首次分配服务器         │ tomcat2               │
│ 会话保持次数           │ 100 次                │
│ 服务器变更次数         │ 0 次                  │
│ 会话保持率             │ 100.00%              │
└────────────────────────────────────────────────┘
```

**[IP哈希算法测试截图]**：左侧显示100次请求全部返回tomcat2的badge；右侧显示响应头中 `X-Upstream-Server: 172.28.0.12:8081` 始终不变。

**分析**：
- 同一客户端IP的所有请求被哈希映射到同一台服务器（tomcat2）
- 实现了**100%的会话保持率**，无需应用层的Session共享机制
- 验证了IP哈希的天然粘性会话能力
- 缺点：如果多个用户通过同一NAT网关访问，所有请求会集中在同一节点

#### 2.4 三种算法对比总结

| 对比维度 | 轮询 | 最少连接 | IP哈希 |
|---------|------|---------|--------|
| **请求分发均匀度** | ⭐⭐⭐⭐⭐ (33/34/33) | ⭐⭐⭐⭐ (36/31/33) | ⭐ (100/0/0单客户端) |
| **会话保持能力** | ❌ 不保持 | ❌ 不保持 | ✅ 100%保持 |
| **响应时间** | 0.012s | 0.010s | 0.009s |
| **配置复杂度** | 低（默认） | 中（加 least_conn;） | 低（加 ip_hash;） |
| **适用场景** | 无状态应用，配置均匀 | 处理时间差异大的请求 | 需要Session粘性的应用 |

#### 2.5 实验结论

1. **轮询是默认首选**：配置最简单，在等权同构集群中分发最均匀
2. **最少连接适合异构场景**：当请求处理时间差异大时能智能避开繁忙节点
3. **IP哈希解决会话问题**：无需应用层改造即可实现会话保持，但可能造成负载不均
4. **生产建议**：使用最少连接算法 + Redis集中式Session存储，兼顾负载均衡与会话管理
