## 一、服务器运维与性能优化核心理论知识

### 1.1 简述整套服务器基础设施运维工作的核心目标

服务器基础设施运维（Server Infrastructure Operations）是现代 IT 服务管理的核心支柱，其核心目标可以从以下六个维度进行系统阐述：

**（一）服务可用性与稳定性保障**

服务可用性是运维工作的首要目标。业界普遍采用 SLA（Service Level Agreement）来衡量，标准商用系统通常要求达到 99.9%（三个九，年停机时间不超过 8.76 小时），金融级系统甚至要求 99.99%（四个九）或 99.999%（五个九）。运维团队需通过冗余架构设计、故障自动切换（Failover）、健康检查探针等手段，确保业务服务持续可访问。Google SRE（Site Reliability Engineering）体系中提出的 Error Budget 概念，将可用性目标量化为可操作的运维指标，是当前行业最佳实践。

**（二）基础设施性能优化**

性能优化是运维工作从"能用"到"好用"的关键跃升。核心在于通过系统化的监控体系，实时采集 CPU、内存、磁盘 I/O、网络带宽、应用响应时间等关键性能指标（KPI），建立性能基线（Baseline），识别瓶颈点。当系统出现性能退化时，能够基于数据驱动的分析方法，精准定位瓶颈层次（硬件层、OS 内核层、应用层、数据库层），实施针对性优化措施，并通过 A/B 测试或金丝雀发布验证优化效果。

**（三）安全与合规**

安全运维涵盖主机安全加固（如 CIS Benchmark 合规检查）、访问控制（最小权限原则）、漏洞扫描与修复、入侵检测与响应、日志审计等方面。在国内语境下，还需符合《网络安全法》、等级保护 2.0（等保 2.0）、《数据安全法》和《个人信息保护法》（PIPL）等法律法规要求。运维人员需建立安全基线，定期开展渗透测试和安全演练，确保基础设施的机密性（Confidentiality）、完整性（Integrity）和可用性（Availability）。

**（四）成本效率与资源优化**

在云计算时代，基础设施成本管理（FinOps）成为运维核心目标之一。运维团队需通过资源使用率分析（如 CPU/内存利用率、存储使用率），识别闲置资源（Idle Resources），实施资源右尺寸（Right-sizing）、预留实例（Reserved Instances）购买、Spot 实例利用等成本优化策略。同时，通过自动化伸缩（Auto Scaling）技术，实现资源供给与业务负载的动态匹配，避免过度配置（Over-provisioning）造成的浪费。

**（五）灾备与业务连续性**

灾难恢复（DR）和业务连续性（BCP）是运维工作的兜底保障。核心目标包括：制定 RTO（Recovery Time Objective，恢复时间目标）和 RPO（Recovery Point Objective，恢复点目标）；建立异地多活或主备切换架构；定期开展灾备演练，验证恢复流程的有效性；确保数据备份的完整性、可恢复性和安全性。典型的 3-2-1 备份策略（3 份副本、2 种介质、1 份异地）仍是行业通行做法。

**（六）自动化与标准化**

通过 Infrastructure as Code（IaC，如 Terraform、Ansible）、CI/CD 流水线（Jenkins、GitLab CI）、容器编排（Kubernetes）等技术手段，将重复性运维操作代码化、流程化，实现环境一致性、部署可重复性、变更可追溯性。标准化则体现在操作系统基线、中间件配置规范、监控指标统一命名等方面，降低人工操作失误风险，提升运维效率。

综上所述，服务器基础设施运维的核心目标是一个多维度的目标体系：**以可用性为底线，以性能优化为动力，以安全合规为保障，以成本控制为约束，以灾备体系为兜底，以自动化标准化为手段**，最终实现 IT 基础设施对业务的高效、稳定、安全支撑。

### 1.2 阐述服务器运维岗日常核心工作内容，并论述运维人员对应的职业道德与数字信息领域社会责任

**（一）日常核心工作内容**

服务器运维岗位的日常工作可以归纳为"监、管、维、优、急"五个维度：

**1. 监控与巡检（Monitoring & Inspection）**

这是运维工作的"眼睛"。每日需通过监控系统（如 Prometheus + Grafana、Zabbix、Datadog）巡检所有服务器的健康状态，包括：
- 基础资源：CPU 使用率、内存使用率、磁盘使用率、网络流量是否在正常范围内
- 服务状态：Web 服务器、数据库、消息队列等核心服务是否正常运行
- 日志分析：通过 ELK（Elasticsearch, Logstash, Kibana）或 Loki 集中日志平台，排查 ERROR/WARN 级别日志
- 告警处理：对监控系统触发的告警进行分级响应（P0 紧急/P1 重要/P2 一般），按 SLA 规定时限处理

**2. 变更管理（Change Management）**

包括应用发布、配置变更、安全补丁更新、系统升级等操作。规范的变更流程需遵循：变更申请 → 影响评估 → 审批 → 灰度发布 → 全量上线 → 回滚预案。所有变更操作应在变更窗口（Change Window）内进行，并在 CMDB（配置管理数据库）中记录。

**3. 故障处理与应急响应（Incident Response）**

当业务出现故障时，运维人员需快速响应：
- 故障发现（Detection）→ 通过监控告警或用户反馈
- 故障定界（Triage）→ 确定影响范围和严重程度
- 故障定位（Diagnosis）→ 通过日志、指标、调用链追踪根因
- 故障恢复（Resolution）→ 执行应急预案（如切流、重启、回滚）
- 复盘改进（Postmortem）→ 编写故障报告，推动长期修复

**4. 容量规划与性能优化（Capacity Planning）**

定期分析资源使用趋势，预测未来容量需求，提前进行资源扩容或架构优化。建立性能基线，当系统性能偏离基线时（如响应时间增加 50%），触发性能优化流程。遵循 USE 方法（Utilization 利用率、Saturation 饱和度、Errors 错误）和 RED 方法（Rate 速率、Errors 错误、Duration 延迟）进行系统化性能分析。

**5. 自动化运维（Automation）**

将重复性操作脚本化，如：自动部署脚本、健康检查脚本、数据备份脚本、日志清理脚本等。使用 Ansible、SaltStack 等配置管理工具实现批量服务器管理。构建 CI/CD 流水线，实现代码提交到生产部署的全流程自动化。

**（二）职业道德与社会责任**

运维人员掌握着企业核心 IT 基础设施的最高权限，其职业道德水准直接关系到企业信息安全和业务连续性。

**1. 数据保密与隐私保护**

运维人员可能接触到数据库中的用户个人信息、业务敏感数据等。必须严格遵守数据访问权限管理制度，做到：非必要不接触、接触不泄露、操作有记录。遵循最小权限原则（Principle of Least Privilege），仅在执行特定运维任务时临时获取所需权限。任何未经授权的数据查询、复制或外传行为都是严重违规。

**2. 诚实守信与职业操守**

运维操作直接影响线上业务，必须实事求是、严禁虚假操作。变更操作不得隐瞒、故障信息不得虚报、操作记录不得伪造。在执行变更时，要严格按照审批流程操作，不得跳过关键审批环节。出现操作失误时，应第一时间上报而非隐瞒，以减少故障影响。

**3. 责任意识与担当精神**

运维工作具有"功在平时、危难时刻见真章"的特点。运维人员需具备强烈的责任心，主动发现和解决潜在风险。面对突发故障，要保持冷静、快速响应、及时沟通，把业务影响降到最低。

**4. 数字信息领域社会责任**

在数字化转型的大背景下，运维人员承担着重要的社会责任：
- **维护网络空间安全**：确保所运维的系统不被黑客入侵、不被用作攻击跳板、不成为数据泄露源头
- **保障公共服务稳定性**：很多运维系统服务于教育、医疗、政务等公共领域，系统稳定性直接关系到社会公众利益
- **推动技术普惠**：积极参与开源社区，贡献运维工具和经验，降低中小企业的运维门槛
- **践行绿色运维**：通过资源优化、虚拟化整合等手段，降低数据中心能耗，助力碳中和目标
- **合法合规运营**：严格遵守《网络安全法》《数据安全法》等法律法规，落实实名制、日志留存、数据跨境传输合规等要求

### 1.3 实际生产环境中，落地服务器运维工作会用到哪些常规实施方法？分点说明

**1. 监控体系建设**

监控是运维的基础设施。生产环境通常采用分层监控架构：
- **基础设施层**：使用 Prometheus + Node Exporter 采集主机 CPU、内存、磁盘、网络指标
- **中间件层**：JMX Exporter (Java)、Redis Exporter、Nginx Exporter 等专用采集器
- **应用层**：APM 工具（如 SkyWalking、Pinpoint、Datadog）实现分布式调用链追踪
- **日志层**：ELK Stack 或 Loki + Grafana 实现集中日志管理和全文检索
- **告警层**：Alertmanager 配合企业微信/钉钉/飞书/PagerDuty 实现分级告警通知

**2. 配置管理与自动化**

- **配置管理**：使用 Ansible Playbook 定义服务器配置基线，确保所有服务器配置一致；使用 Puppet 或 SaltStack 实现配置的自动推送和合规检查
- **基础设施即代码（IaC）**：Terraform 管理云资源（虚拟机、网络、存储），HCL 声明式配置实现环境快速复制；Packer 构建标准化镜像
- **CI/CD**：Jenkins Pipeline 或 GitLab CI 定义构建-测试-部署流水线；Helm Chart 管理 Kubernetes 应用部署

**3. 容器化与编排**

- Docker 镜像标准化应用运行环境，消除"在我机器上能跑"的环境差异
- Kubernetes 提供容器编排、服务发现、负载均衡、滚动更新、自动伸缩等能力
- 服务网格（如 Istio）实现微服务间的流量管理、安全通信和可观测性

**4. 日志管理与分析**

- Filebeat/Fluentd 采集日志 → Kafka 消息队列缓冲 → Logstash 处理 → Elasticsearch 存储 → Kibana 展示
- 对于容器化环境，使用 Loki（轻量级、与 Grafana 深度集成）+ Promtail 采集
- 日志标准化：统一使用 JSON 格式，包含 timestamp、level、service、trace_id 等字段

**5. 备份与灾备**

- 数据库：全量备份（每日）+ 增量备份（每小时）+ binlog 实时同步
- 文件存储：rsync 或对象存储跨区域复制
- 灾备架构：同城双活 + 异地灾备；定期灾备切换演练（每季度至少一次）
- 备份验证：定期从备份恢复数据到测试环境，验证备份可用性

**6. 安全运维**

- 主机安全：定期漏洞扫描（Nessus/OpenVAS），及时安装安全补丁
- 网络安全：防火墙策略最小化、入侵检测（Snort/Suricata）、DDoS 防护
- 访问控制：堡垒机统一管理运维入口，SSH 密钥认证 + 多因素认证
- 合规审计：所有运维操作通过堡垒机录屏/录命令，日志保留不少于 6 个月

**7. 文档化与知识管理**

- 建立运维知识库（Wiki/Confluence），沉淀故障案例、操作手册、最佳实践
- CMDB 维护 IT 资产信息和配置项关系
- Runbook（运维操作手册）覆盖所有标准运维操作流程

**8. 持续优化与成本管理**

- 定期性能评估（月度/季度），生成资源使用报告和优化建议
- FinOps 实践：标记资源归属、分析成本构成、识别优化机会
- 容量规划：基于历史数据预测未来 3-6 个月资源需求

### 1.4 开展服务器性能监控、性能调优工作，需要覆盖哪些层级、哪些具体对象？通用实施思路是什么？

**（一）性能监控需要覆盖的层级与对象**

服务器性能监控应采用自底向上的全栈覆盖思路，从硬件到用户体验逐层监控：

**第一层：硬件层**

- 监控对象：CPU 温度、风扇转速、内存 ECC 错误、磁盘 SMART 数据、网卡状态、电源状态
- 工具：ipmitool、lm-sensors、smartctl
- 关键指标：硬件健康状态、温度是否超过阈值、是否有硬件级错误

**第二层：操作系统内核层**

- 监控对象：
  - CPU：利用率、运行队列长度、上下文切换率、中断频率
  - 内存：使用率、页面错误（page fault）、Swap 使用、OOM 事件
  - 磁盘 I/O：IOPS、吞吐量、延迟、利用率、队列深度
  - 网络：带宽利用率、丢包率、错误包数、连接数
  - 文件系统：inode 使用率、文件描述符使用数
- 工具：eBPF、perf、iostat、vmstat、sar、/proc 文件系统

**第三层：网络层**

- 监控对象：网络延迟、丢包率、带宽利用率、TCP 重传率、连接状态分布
- 工具：ping、mtr、iperf3、ss、tcpdump

**第四层：应用服务层**

- 监控对象：Web 服务器（Nginx/Apache）、应用服务器（Tomcat/Node.js）、消息队列（Kafka/RabbitMQ）、缓存（Redis/Memcached）
- 关键指标：
  - 请求量（QPS/RPS）、响应时间（P50/P95/P99）
  - 错误率（5xx/4xx 比例）
  - 线程池/连接池使用率
  - GC（垃圾回收）频率和暂停时间（Java）
  - 消息积压量（消息队列）

**第五层：数据库层**

- 监控对象：MySQL/PostgreSQL/MongoDB
- 关键指标：
  - 慢查询数量和执行时间
  - 连接数使用率
  - 缓冲池命中率（Buffer Pool Hit Rate）
  - 主从复制延迟
  - 锁等待和死锁

**第六层：用户体验层**

- 监控对象：页面加载时间、首屏渲染时间、API 响应时间、可用性
- 工具：前端性能监控（RUM，Real User Monitoring）、合成监控

**（二）通用性能调优实施思路**

性能调优不是一次性的活动，而是持续循环的过程。通用思路可以归纳为"观察→测量→分析→优化→验证→循环"的六步方法论：

**第一步：建立性能基线（Baseline）**

在进行任何优化之前，必须建立当前的性能基线。记录系统在正常负载下的各项性能指标，包括 QPS、响应时间 P50/P95/P99、CPU/内存使用率等。基线数据是判断优化是否有效的参照物。

**第二步：监控与发现（Monitoring）**

部署全面的监控体系，使用 USE 方法（Utilization 利用率、Saturation 饱和度、Errors 错误）分析每个资源：

- **利用率**：资源被使用的百分比。CPU 90% 利用率说明可能是瓶颈
- **饱和度**：等待资源的队列长度。CPU run queue > 核数说明过载
- **错误**：发生的错误次数。网卡 CRC 错误说明可能有硬件问题

对于服务层，使用 RED 方法（Rate 请求速率、Errors 错误率、Duration 延迟）：

- **Rate**：每秒请求数
- **Errors**：失败请求比例
- **Duration**：请求处理时间的分布

**第三步：定位瓶颈（Identification）**

这是最关键的步骤。通过排除法从上到下或从下到上逐层排查：
- 检查硬件层（是否有硬件故障？）
- 检查 OS 层（系统参数是否合理？资源是否饱和？）
- 检查网络层（是否有丢包？延迟是否过高？）
- 检查应用层（是否有慢请求？是否有资源泄漏？）
- 检查数据库层（是否有慢查询？是否有锁竞争？）

使用 80/20 法则：80% 的性能问题由 20% 的瓶颈引起。优先解决影响最大的瓶颈。

**第四步：实施优化（Optimization）**

根据瓶颈类型选择优化策略：

- **CPU 瓶颈**：代码优化（算法改进）、并行化、CPU 亲和性绑定、升级 CPU
- **内存瓶颈**：减少内存分配、使用对象池、调整 GC 参数、增加内存
- **磁盘 I/O 瓶颈**：使用 SSD、RAID 优化、文件系统选择（XFS/ext4）、I/O 调度器调整
- **网络瓶颈**：TCP 参数优化、使用连接池、CDN 加速、增加带宽
- **应用瓶颈**：缓存优化（多级缓存）、数据库查询优化（索引、SQL 改写）、异步化

**第五步：验证效果（Validation）**

优化后必须验证效果：
- 将优化后的性能指标与基线对比
- 使用相同的负载工具和场景进行压力测试
- 确认优化没有引入新的问题或回归
- 通过 A/B 测试或金丝雀发布逐步验证

**第六步：持续监控（Continuous Monitoring）**

优化不是终点。建立持续的性能监控机制，设置性能退化告警。当性能指标偏离预期范围时，自动触发新一轮的性能分析。将性能测试纳入 CI/CD 流水线，在代码上线前自动验证性能影响。

整个过程的本质是 OODA 循环（Observe 观察 → Orient 判断 → Decide 决策 → Act 行动）在性能领域的应用，核心在于**数据驱动、精准优化、持续迭代**。

### 2.3 Nginx 负载均衡算法切换对比实验

#### 2.3.1 三种算法对比分析

| 算法 | 分发策略 | 适用场景 | 优点 | 缺点 |
|------|----------|----------|------|------|
| Round-Robin (轮询) | 按顺序轮流分发 | 后端服务器性能一致 | 配置简单，分发均匀 | 不考虑服务器实际负载 |
| Least Connections (最少连接) | 分发给活跃连接数最少的服务器 | 请求处理时间不均匀 | 动态适配负载 | 需要 Nginx 维护连接状态 |
| IP Hash | 对客户端 IP 取哈希，同一 IP 始终到同一服务器 | 需要 Session 保持 | 无需共享 Session | 负载分发不均匀 |

#### 2.3.2 算法切换方法

使用提供的 `switch-algorithm.sh` 脚本一键切换：

```bash
cd task2-cluster/nginx
bash switch-algorithm.sh round-robin   # 切换到轮询
bash switch-algorithm.sh least-conn    # 切换到最少连接
bash switch-algorithm.sh ip-hash       # 切换到 IP Hash
```

每次切换后，使用压力测试脚本验证分发效果：

```bash
cd task2-cluster/stress-test
bash test.sh all   # 遍历三种算法并对比
```

#### 2.3.3 实验结论

- **Round-Robin**：多次请求均匀分发至三台服务器，适合无状态应用
- **Least Connections**：在请求处理时间不均匀时，负载更均衡
- **IP Hash**：同一客户端始终命中同一服务器，适合有状态应用但可能导致负载倾斜

## 三、基于 eBPF 的 Linux 性能观测工具设计与实现

### 3.1 项目背景与意义

随着云计算和大数据产业的快速发展，Linux 操作系统凭借其开源、稳定、高性能的特点，成为服务器、云平台和大数据集群的核心操作系统。精准采集和分析 Linux 系统的底层性能数据，是排查系统故障、开展性能调优、保障业务稳定运行的关键。

传统的 Linux 性能监测方法（如 top、vmstat、iostat 等）主要在用户空间采集数据，存在以下局限性：
- **运行开销大**：需要频繁读取 /proc 文件系统，系统调用开销高
- **监控维度少**：只能看到粗略的聚合数据，缺乏细粒度指标
- **无法深入内核**：无法观测内核调度器、内存分配器、VFS 层等核心子系统的内部状态
- **采样精度低**：基于轮询的采集方式，可能错过短时突发事件

eBPF (extended Berkeley Packet Filter) 技术作为新一代内核级可编程技术，完美解决了上述问题。它允许在不修改内核源码、无需重启系统的情况下，在内核空间安全地运行沙箱化的程序，具有以下优势：
- **高性能**：eBPF 程序在内核态执行，通过 JIT 编译为本地机器码，几乎没有性能损耗
- **零侵入**：通过 tracepoint、kprobe、fentry 等机制动态挂载，无需修改内核
- **高精度**：可以捕获每次系统调用、每次调度事件、每次网络包收发
- **安全**：eBPF 验证器确保程序不会导致内核崩溃

### 3.2 系统架构设计

本系统采用分层架构：**内核态 eBPF 数据采集层 + 用户态数据处理层 + 交互展示层**。

```
┌──────────────────────────────────────────────────────────────────┐
│                    交互展示层 (Python)                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────────────┐ │
│  │ 终端 UI  │  │ CSV 存储 │  │ 数据库   │  │ Prometheus 端点  │ │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────────┬─────────┘ │
├───────┼─────────────┼─────────────┼──────────────────┼───────────┤
│       │        用户态数据处理层 (C)                      │           │
│  ┌────┴─────┐ ┌────┴─────┐ ┌────┴─────┐ ┌────┴─────┐ ┌──┴────────┐│
│  │  CPU     │ │  Memory  │ │  Disk    │ │  File    │ │  Network  ││
│  │  Loader  │ │  Loader  │ │  Loader  │ │  Loader  │ │  Loader   ││
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬──────┘│
│       │            │            │            │            │        │
│       │    BPF Maps (Ring Buffer / Hash / Array)            │        │
├───────┼────────────┼────────────┼────────────┼────────────┼────────┤
│       │        内核态 eBPF 采集层                             │        │
│  ┌────┴─────┐ ┌────┴─────┐ ┌────┴─────┐ ┌────┴─────┐ ┌──┴────────┐│
│  │ sched/   │ │ kmem/    │ │ block/   │ │ syscalls/│ │ net/      ││
│  │ sched_*  │ │ oom/     │ │ block_*  │ │ sys_*    │ │ tcp/      ││
│  │ power/   │ │ page_*   │ │          │ │          │ │ net_dev_* ││
│  └─────────┘ └──────────┘ └──────────┘ └──────────┘ └───────────┘│
├──────────────────────────────────────────────────────────────────┤
│                        Linux Kernel 6.17                          │
└──────────────────────────────────────────────────────────────────┘
```

**数据流**：
1. eBPF 程序挂载在内核 tracepoint/kprobe 上
2. 内核事件触发 eBPF 程序执行，采集性能数据
3. 数据通过 BPF Ring Buffer 高效传输到用户态
4. 用户态 C 程序读取数据、格式化输出，同时写入 CSV 文件和数据库
5. Python UI 提供交互式菜单、日志查看、数据导出等功能
6. Prometheus HTTP 端点提供指标供 Grafana 可视化

### 3.3 关键性能指标设计与实现

本项目共实现了 **20 个关键性能指标**，覆盖 5 大模块：

#### 3.3.1 CPU 性能监测（5 个指标）

**实现原理**：通过挂载在 `sched/sched_switch` 和 `sched/sched_wakeup` 的 eBPF 程序捕获每次任务切换事件，统计上下文切换次数和运行队列深度。挂载在 `power/cpu_frequency` 的 eBPF 程序捕获 CPU 调频事件。用户态程序读取 `/proc/stat` 计算各 CPU 核心的利用率。

| 序号 | 指标名称 | 采集方式 | eBPF 挂载点 |
|------|----------|----------|-------------|
| 1 | CPU 利用率 (user%/system%/idle%/iowait%) | eBPF + /proc/stat | tracepoint/sched/sched_switch |
| 2 | 运行队列长度 | eBPF | tracepoint/sched/sched_wakeup |
| 3 | 上下文切换率 (次/秒) | eBPF | tracepoint/sched/sched_switch |
| 4 | CPU 频率 (MHz) | eBPF | tracepoint/power/cpu_frequency |
| 5 | CPU 空闲时间分布 | /proc/stat 分析 | 用户态计算 |

#### 3.3.2 内存性能监测（5 个指标）

**实现原理**：通过挂载在 `exceptions/page_fault_user` 的 eBPF 程序统计页面错误次数。挂载在 `kmem/kmalloc` 的 eBPF 程序跟踪内核内存分配。挂载在 `oom/oom_kill_process` 捕获 OOM 事件。用户态程序读取 `/proc/meminfo` 获取内存使用概况。

| 序号 | 指标名称 | 采集方式 | eBPF 挂载点 |
|------|----------|----------|-------------|
| 1 | 内存使用率 (Total/Used/Free/Cached) | /proc/meminfo 分析 | 用户态采集 |
| 2 | 页面错误率 (Major/Minor faults/sec) | eBPF | tracepoint/exceptions/page_fault_user |
| 3 | Swap 使用量 | /proc/meminfo 分析 | 用户态采集 |
| 4 | OOM 事件计数 | eBPF | tracepoint/oom/oom_kill_process |
| 5 | 内核内存分配率 (kmalloc bytes/sec) | eBPF | tracepoint/kmem/kmalloc |

#### 3.3.3 磁盘 I/O 性能监测（5 个指标）

**实现原理**：通过挂载在 `block/block_rq_issue` 记录 I/O 请求发起时间，在 `block/block_rq_complete` 计算 I/O 延迟并更新统计。用户态程序读取 `/proc/diskstats` 获取各磁盘设备的 IOPS 和吞吐量。

| 序号 | 指标名称 | 采集方式 | eBPF 挂载点 |
|------|----------|----------|-------------|
| 1 | 磁盘读取字节/秒 | eBPF + /proc/diskstats | tracepoint/block/block_rq_complete |
| 2 | 磁盘写入字节/秒 | eBPF + /proc/diskstats | tracepoint/block/block_rq_complete |
| 3 | IOPS (读写操作数/秒) | /proc/diskstats 分析 | tracepoint/block/block_rq_issue |
| 4 | I/O 平均延迟 (微秒) | eBPF 计算 | block_rq_issue → block_rq_complete |
| 5 | 磁盘利用率 (%) | /proc/diskstats io_ticks | 用户态计算 |

#### 3.3.4 文件 I/O 性能监测（5 个指标）

**实现原理**：通过挂载在 syscall tracepoint (`sys_enter_openat`, `sys_enter_close`, `sys_enter_read`, `sys_enter_write`, `sys_enter_fsync`) 的 eBPF 程序捕获文件操作。用户态程序读取 `/proc/sys/fs/dentry-state` 和 `/proc/sys/fs/inode-nr` 获取文件系统缓存状态。

| 序号 | 指标名称 | 采集方式 | eBPF 挂载点 |
|------|----------|----------|-------------|
| 1 | 文件打开/关闭次数 | eBPF | tracepoint/syscalls/sys_enter_openat/close |
| 2 | VFS 读取次数 | eBPF | tracepoint/syscalls/sys_enter_read |
| 3 | VFS 写入次数 | eBPF | tracepoint/syscalls/sys_enter_write |
| 4 | 目录项缓存 (dentry) 条目数 | /proc/sys/fs/dentry-state | 用户态采集 |
| 5 | Inode 缓存分配/空闲数 | /proc/sys/fs/inode-nr | 用户态采集 |

#### 3.3.5 网络性能监测（5 个指标）

**实现原理**：通过挂载在 `net/net_dev_queue`（发包）、`net/netif_receive_skb`（收包）、`net/net_dev_xmit`（发包完成/错误）和 `tcp/tcp_retransmit_skb`（TCP 重传）的 eBPF 程序捕获网络事件。用户态程序读取 `/proc/net/dev` 和 `/proc/net/tcp` 获取每接口统计和 TCP 连接状态。

| 序号 | 指标名称 | 采集方式 | eBPF 挂载点 |
|------|----------|----------|-------------|
| 1 | 网络收发字节/秒 | eBPF + /proc/net/dev | tracepoint/net/netif_receive_skb + /proc/net/dev |
| 2 | 网络收发包/秒 | /proc/net/dev 分析 | tracepoint/net/net_dev_queue |
| 3 | TCP 连接状态分布 | /proc/net/tcp 分析 | 用户态采集 |
| 4 | 网络错误/丢包数 | eBPF + /proc/net/dev | tracepoint/net/net_dev_xmit |
| 5 | TCP 重传次数 | eBPF | tracepoint/tcp/tcp_retransmit_skb |

- TCP 重传率高 → 检查网络质量、调整 TCP 参数

### 3.8 技术难点与解决方案

1. **eBPF 验证器限制**：eBPF 程序在内核态运行，需经过严格的验证器（Verifier）检查。循环必须有编译时确定的上界，程序大小和复杂度受限。解决：合理设计 BPF 程序逻辑，避免复杂分支，将复杂计算下移到用户态。

2. **CO-RE 兼容性**：不同内核版本的数据结构可能不同。通过 BTF（BPF Type Format）生成 `vmlinux.h`，使用 BPF CO-RE 宏实现一次编译、随处运行。

3. **性能开销控制**：eBPF 程序在内核热路径上执行（如每次系统调用），必须极度高效。使用 per-CPU 数据结构和无锁操作避免竞争，通过采样而非全量处理减少开销。

4. **数据安全传输**：内核态到用户态的数据传输使用 Ring Buffer（替代旧的 perf buffer），支持内存映射、减少拷贝。
